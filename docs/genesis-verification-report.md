## Overview

This document provides an example of the verification audit report generated by the `scripts/verify-clean-genesis.js` tool. The tool is designed to detect and prevent accidental "genesis" references throughout the MOBIUS repository.

## Verification Scope

The genesis verification tool scans the following areas:

### 1. Working Tree Files
- All source code files (`.js`, `.ts`, `.json`, `.md`, etc.)
- Configuration files
- Documentation files
- Excludes node_modules, build artifacts, and the verification tools themselves

### 2. Git History
- Commit messages across all branches
- Diff content from all commits
- Searches through the complete git history

### 3. Temporary/Backup Files
- `.tmp`, `.temp`, `.bak`, `.backup` files
- Editor swap files (`.swp`, `.swo`)
- Files in `/tmp/` and `/temp/` directories
- Backup files with `~` suffix

### 4. Binary Blob Scan (Detailed Mode Only)
- Binary files that may contain embedded text
- Images with embedded metadata
- Compiled artifacts
- Archive files

## Exclusion Patterns

The following patterns are automatically excluded from scanning to avoid false positives:

- `scripts/verify-clean-genesis.js` - The verification script itself
- `docs/genesis-verification-report.md` - This documentation
- `verification-reports/` - Generated report directories
- `.git/objects/` - Git internal objects
- `node_modules/` - Dependencies
- `coverage/`, `dist/`, `build/` - Build artifacts
- `.DS_Store`, `Thumbs.db` - OS files
- `*.log` - Log files

## Integration with CI/CD

### GitHub Actions Example
```yaml
name: Genesis Verification

on:
  pull_request:
    branches: [ main ]
  push:
    branches: [ main ]

jobs:
  genesis-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with:
          fetch-depth: 0  # Full history for git scanning
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          
      - name: Install dependencies
        run: npm ci
        
      - name: Run Genesis Verification
        run: node scripts/verify-clean-genesis.js --ci
        
      - name: Upload verification report
        if: failure()
        uses: actions/upload-artifact@v3
        with:
          name: genesis-verification-report
          path: verification-reports/
```

### Jenkins Pipeline Example
```groovy
pipeline {
    agent any
    stages {
        stage('Genesis Verification') {
            steps {
                sh 'npm ci'
                sh 'node scripts/verify-clean-genesis.js --ci'
            }
            post {
                always {
                    archiveArtifacts artifacts: 'verification-reports/**/*', 
                                   allowEmptyArchive: true
                }
                failure {
                    emailext (
                        subject: "Genesis verification failed: ${env.JOB_NAME} - ${env.BUILD_NUMBER}",
                        body: "Genesis references detected. Check the verification report.",
                        to: "${env.CHANGE_AUTHOR_EMAIL}"
                    )
                }
            }
        }
    }
}
```

## Exit Codes

- **0**: Repository is clean (no genesis references found)
- **1**: Genesis references detected (CI job should fail)

## Report Generation

Each verification run generates a timestamped report in:
```
verification-reports/YYYYMMDD_HHMMSS/genesis-verification-report.md
```

Reports include:
- Detailed match information with file paths and line numbers
- Context around each match
- Categorization by scan type
- Recommendations for remediation

## Security Considerations

- The tool intentionally excludes its own files to prevent self-matches
- Binary scanning is optional due to performance impact
- Git history scanning requires full repository clone
- Reports may contain sensitive file paths - review before sharing

## Troubleshooting

### Common Issues

1. **"Could not scan git history"** - Not a git repository or git not installed
2. **Permission errors** - Check file system permissions for temp directories
3. **High memory usage** - Use fast mode instead of detailed for large repositories

### Performance Tips

- Use fast mode (`--ci`) for regular CI checks
- Run detailed mode (`--detailed`) periodically or on-demand
- Exclude large binary directories if they don't need scanning
